## 交叉引用

- `leak-audit` 覆盖了本 skill 未深入探讨的错误响应格式指纹、日志泄漏、JA4S 服务端指纹、部署规模追踪维度
- `traffic-audit` 覆盖了本 skill 未深入探讨的流量统计分析、包长分布、时间间隔模式、多会话关联分析维度
- `crypto-audit` 覆盖了本 skill 未深入探讨的 AEAD 密钥管理、HKDF 密钥派生、证书与签名维度

## 审计流程

1. **识别变更范围**: 确定本次修改涉及哪个伪装方案、哪个握手阶段（ClientHello 解析 / ServerHello 构造 / 应用层数据处理）
2. **追踪握手路径**: 从 ClientHello 接收到最终响应发送，追踪完整握手路径上的所有状态转换
3. **计算 JA4+ 指纹**: 使用 JA4 算法计算变更后的 ClientHello 指纹，使用 JA4S 计算 ServerHello 指纹，与目标浏览器的已知指纹对比
4. **交叉一致性验证**: SNI → 证书 CN/SAN → 证书签发链 → ALPN → 后续流量类型 → TCP/IP 栈特征 → GREASE 值 → 后量子扩展 → TLS 库行为，整条链是否逻辑自洽
5. **状态机完整性**: 验证所有状态转换（包括正常路径、错误路径、降级路径）与目标实现一致
6. **握手后指纹**: 验证应用层帧序列（HTTP/2 SETTINGS → SETTINGS ACK → HEADERS 等）与主流实现一致
7. **TLS 库行为审计**: 确认服务端的密码套件选择偏好、Alert 描述码、记录构造、OCSP/Certificate Compression 支持状态与声称的 TLS 库一致
8. **TCP/IP 层验证**: 确认 TTL、窗口大小、TCP 选项顺序与 TLS 指纹声称的操作系统一致
9. **测试回落机制**: 模拟非预期输入（空 SNI、不支持的 ALPN、协议降级请求），确认回落行为安全无异常
10. **方案隔离测试**: 如果修改涉及方案选择逻辑，验证方案之间的指纹隔离和缓冲区边界

## 指纹更新维护建议

指纹伪装是一个**持续对抗**的过程，不是一次性工作。以下维护策略应纳入开发流程：

- **版本跟踪**: 为每个指纹模板标注目标浏览器版本号和采集日期
- **变更检测**: 监控 Chrome 每个大版本（约 4 周一个）的指纹变化，重点关注扩展新增/删除/顺序变化
- **自动化验证**: 建立指纹回归测试，每次修改后自动计算 JA4+ 指纹并与基线对比
- **多版本覆盖**: 考虑同时支持多个浏览器版本的指纹模板，随机选择使用哪个版本，增加指纹多样性
- **降级监控**: 定期验证所有活跃指纹模板的连通率，及时淘汰被审查系统识别的旧模板
