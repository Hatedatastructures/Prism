# Skill: 信息泄漏审计

在修改涉及协议处理、日志输出、错误响应、配置文件、证书生成代码后，必须对变更部分执行以下审计清单。审查系统通过收集代理软件泄漏的唯一标识符、版本字符串、错误格式、二进制格式规律等指纹来建立黑名单。

## 审查系统的检测原理

审查系统的信息收集是多层次的：

1. **字符串指纹**：协议交互中的软件标识（如 `server=xxx`）、版本字符串、自定义错误格式。这些是确定性的黑名单匹配。
2. **二进制格式指纹**：首包的固定长度字段结构（如认证 hash + 地址信息的固定偏移）。即使加密后，固定长度的首包也能被 ML 训练提取。
3. **错误行为指纹**：非标准输入触发的错误响应格式。标准 Web 服务器返回标准 HTTP/TLS 错误，代理软件返回的任何独特格式都是指纹。
4. **网络栈指纹**：TLS 扩展顺序、HTTP/2 Settings 参数、TCP 窗口大小等协议参数。这些参数的特定组合本身就是客户端/服务端的身份标识。
5. **版本/实现特征**：软件版本升级后行为变化（如新增扩展、参数微调）可被用来追踪软件版本和部署规模。
6. **实现独特行为**：每种代理软件实现都有独特的行为指纹 — 帧格式细节、错误处理路径、内存分配模式导致的响应时间分布。即使没有字符串泄漏，实现层面的独特行为也可以建立"行为指纹库"。
7. **开源代码指纹**：审查系统可以分析开源代理软件的代码，提取其实现的独特行为模式。如果代码公开，审查系统可以精确到函数级别的行为预测。

## 触发条件

- 修改了任何包含字符串常量的源文件
- 修改了日志输出
- 修改了错误响应格式（HTTP 400/407、SOCKS5 错误等）
- 修改了协议握手或认证响应
- 修改了配置文件
- 修改了证书生成或签名逻辑
- 添加了新的 header 或 HTTP 响应字段
- 修改了首包认证数据格式
- 修改了网络栈参数（TLS 扩展、HTTP/2 Settings、TCP 参数）

## 审计清单

### 1. 协议指纹字符串

| 检查项 | 说明 |
|--------|------|
| **server 标识** | 协议交互中是否包含可识别的软件标识（如 `server=xxx`）？ |
| **版本字符串** | 是否在协议交互中暴露软件版本号？版本字符串是已知的泄漏风险 |
| **错误消息格式** | 错误响应是否使用与主流软件一致的格式？自研格式本身就是指纹 |
| **默认端口/路径** | 是否使用常见的代理默认端口或路径？ |

### 2. 首包格式指纹

| 检查项 | 说明 |
|--------|------|
| **固定长度模式** | 首包是否为固定长度？固定长度（如总是 56 字节）本身就是高信息量特征，可通过 ML 训练提取 |
| **字段分界可识别** | 首包内部是否有明显的字段分界（如密码 hash + 目标地址）？即使加密后，固定偏移处的字节分布也可能被统计提取 |
| **填充破坏模式** | 是否通过随机填充破坏首包的固定长度模式？ |
| **首包与后续包区分度** | 首包与后续数据包的长度/格式是否有明显区分？区分度越高，越容易被识别 |

### 3. 版本/升级追踪防御

| 检查项 | 说明 |
|--------|------|
| **行为一致性** | 软件升级后，协议行为是否保持一致？如果某次升级改变了 TLS 扩展顺序或帧格式，审查系统可以通过行为变化追踪软件版本和部署规模 |
| **配置驱动行为** | 协议行为是否由配置驱动而非硬编码？这样不同部署可以有不同的行为特征，避免全网统一指纹 |
| **去标识化** | 配置协商中是否包含可追踪的版本信息？即使不暴露具体版本号，版本号的存在（如 `v=2`）本身也是版本特征 |

### 4. 实现独特行为去标识

| 检查项 | 说明 |
|--------|------|
| **错误路径独特性** | 审查系统可以构造特定的错误触发条件，不同代理实现在同一条件下的错误响应各不相同。如果错误响应的行为模式可唯一映射到某个已知实现，实现身份暴露 |
| **边界条件行为** | 在协议规范的灰色地带（如未定义的行为），不同实现的处理方式不同。审查系统可以探测这些边界条件来识别实现 |
| **性能特征** | 不同实现在处理特定大小/数量的请求时的性能特征（响应时间分布、内存使用模式）是可区分的 |
| **代码混淆** | 如果代码开源，审查系统可以直接从代码中提取行为模式。关键路径的行为是否可以通过代码分析精确预测？考虑在关键路径引入不可预测的随机化 |

### 5. 证书元数据泄漏

| 检查项 | 说明 |
|--------|------|
| **证书 CN/SAN** | 伪造证书的 Common Name 和 Subject Alternative Name 是否与目标域名完全匹配？不得包含代理软件名称等字样 |
| **证书颁发者** | 颁发者字段是否看起来与真实 CA 一致？ |
| **证书有效期** | 有效期是否与真实网站的证书格式一致？异常短或异常长的有效期是可疑的 |
| **序列号** | 序列号是否随机生成？递增或固定的序列号会暴露批量生成行为 |
| **扩展字段** | Key Usage、Extended Key Usage 等扩展是否完整？缺失扩展是异常特征 |
| **证书透明度日志** | 证书是否被记入 CT Log？CT Log 中的证书注册模式（时间、频率、域名关联）可以被事后分析 |
| **证书签名算法** | 签名算法是否与目标域名的真实证书一致？如果真实证书使用 ECDSA 而伪造证书使用 RSA，算法差异是指纹 |

### 6. 日志安全

| 检查项 | 说明 |
|--------|------|
| **密钥不记录** | 日志中是否包含密钥材料（auth_key、shared_secret、private_key）？ |
| **用户数据不记录** | 日志中是否包含用户密码、请求内容等敏感信息？ |
| **IP 地址脱敏** | 日志中的客户端/目标 IP 是否需要脱敏？ |
| **日志级别控制** | Debug 日志是否仅在开发环境启用？生产环境不得输出调试级别的详细协议信息 |
| **SNI 记录** | 日志中记录的 SNI 是否包含用户访问的真实目标域名？ |
| **日志格式统一** | 日志格式是否与标准服务器软件一致？自定义日志格式如果外泄可能成为指纹 |
| **日志时间戳精度** | 日志时间戳的精度是否过高？毫秒级精度的协议事件时间戳可能泄露处理时间侧信道 |

### 7. 错误响应特征

| 检查项 | 说明 |
|--------|------|
| **HTTP 错误格式** | HTTP 错误响应是否使用标准格式？不得包含自定义 header 或 body 标识 |
| **协议错误** | 协议错误响应是否严格遵循对应 RFC 格式？ |
| **TLS Alert** | TLS Alert 消息是否使用标准错误码？自定义错误码会暴露代理实现 |
| **连接重置** | 异常断开时是否发送标准 TCP RST 而非自定义错误帧？ |
| **错误响应时间** | 错误响应的延迟是否与标准服务器一致？异常快或异常慢的响应都可能是指纹 |
| **错误页面内容** | 错误响应的 body 内容是否与标准服务器（如 Nginx/Apache）一致？空 body 或自定义 HTML 是指纹 |
| **错误响应一致性** | 相同类型的错误是否总是返回完全相同的响应（包括 header 顺序、空格、换行）？微小的格式差异（如 header 顺序）是服务端软件指纹 |

### 8. 配置文件安全

| 检查项 | 说明 |
|--------|------|
| **凭据不硬编码** | 配置文件中的密码、密钥是否以哈希形式存储？不得明文存储 |
| **配置文件权限** | 配置文件是否设置了适当的文件权限？ |
| **默认配置** | 默认配置是否包含示例凭据？用户可能忘记修改 |
| **配置外泄** | 配置文件是否已加入版本控制忽略列表？不得将凭据提交到版本控制 |

### 9. 网络栈指纹

| 检查项 | 说明 |
|--------|------|
| **TLS 库版本** | 使用的 TLS 库版本是否与主流浏览器使用的版本一致？ |
| **TCP 窗口大小** | TCP 初始窗口大小是否与正常 Web 服务器一致？ |
| **TLS 扩展顺序** | TLS 扩展的排列顺序是否与目标浏览器模板完全一致？ |
| **HTTP/2 Settings** | HTTP/2 SETTINGS 帧的参数值是否与主流实现一致？ |
| **TCP/IP 栈参数** | TTL、TCP 选项顺序、DF 位等是否与声称的操作系统一致？TLS 指纹和 TCP/IP 栈指纹的交叉矛盾会暴露伪装 |

### 10. 帧格式泄漏

| 检查项 | 说明 |
|--------|------|
| **帧头魔数** | 自定义帧格式是否包含可识别的魔数或固定前缀？ |
| **命令编码** | 帧命令字段的编码值是否有规律？不应与已知协议的值碰撞 |
| **stream_id 范围** | stream_id 是否从合理范围开始？ |

### 11. 部署规模追踪防御

| 检查项 | 说明 |
|--------|------|
| **去关联化** | 不同部署实例的行为是否足够不同？如果所有部署实例共享完全相同的指纹（TLS 扩展顺序、HTTP/2 参数、错误页面），审查系统可以统计全网部署规模 |
| **实例唯一性** | 每个部署实例是否可以生成独立的随机化参数（如随机化的 HTTP/2 Settings、TLS 扩展顺序微调）？ |
| **全局指纹避免** | 是否存在任何"所有部署实例都相同"的标识？审查系统可以通过全局指纹的出现频率估算部署规模 |

## 审计流程

1. **扫描字符串常量**: 在变更文件中搜索所有硬编码的字符串，特别关注协议交互中的标识符
2. **分析首包格式**: 检查首包的长度和字段结构，评估是否能被 ML 统计提取
3. **评估实现独特性**: 分析错误路径、边界条件行为是否可唯一映射到当前实现
4. **评估版本追踪风险**: 如果升级改变了协议行为，评估是否暴露了版本信息和部署规模
5. **检查日志输出**: 逐行检查新增/修改的日志调用，确认不包含敏感信息
6. **审查错误响应**: 确认所有错误响应使用标准格式，延迟和内容与标准服务器一致，header 顺序固定
7. **验证证书质量**: 如果涉及证书生成，确认伪造证书与真实证书不可区分（包括签名算法一致）
8. **检查网络栈参数**: 确认 TLS 扩展顺序、HTTP/2 Settings、TCP 参数与主流实现一致且内部不矛盾
9. **评估部署规模风险**: 检查是否存在全局统一的指纹，评估审查系统是否能统计全网部署规模
