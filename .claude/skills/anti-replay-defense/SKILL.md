# Skill: 反重放攻击防御审计

在修改涉及协议认证、会话建立、密码验证、首包处理代码后，必须对变更部分执行以下审计清单。审查系统通过录制并重放合法认证包来检测代理服务器，也会利用流加密的延展性实施选择密文攻击。

## 审查系统的检测原理

审查系统的重放攻击是一个**两阶段过程**：

1. **捕获阶段**：被动嗅探发现可疑连接后，从真实客户端捕获合法的握手/认证包（如首包的密码 hash、加密认证数据）
2. **重放阶段**：在数小时甚至**数天后**，向同一服务器重发捕获的包。如果服务器返回加密流量（而非标准 Web 响应），审查系统通过**响应长度和行为**确认目标为代理

关键洞察：重放防御不仅要防止"即时重放"（同一 nonce 重复使用），还要防止"延迟重放"（数天后的重放）。这要求 nonce/序列号的记忆具有一定的持久性。

审查系统还会利用**选择密文攻击**：发送 256 个连接，每个连接在原始捕获包的基础上改变一个字节。如果使用无认证的流加密，某个连接碰巧使得服务端不立即断开，审查系统即可确认该位置的有效性并逐步推断密钥流。AEAD 的认证标签可以彻底阻止此类攻击。

## 触发条件

- 修改了协议认证逻辑（SOCKS5/Trojan/SS2022 等）
- 修改了会话建立流程
- 修改了 AEAD nonce 管理逻辑
- 修改了首包认证处理
- 修改了 nonce/序列号记忆机制

## 审计清单

### 1. AEAD 认证完整性

| 检查项 | 说明 |
|--------|------|
| **认证加密强制** | 是否所有加密通道都使用 AEAD（AES-GCM/ChaCha20-Poly1305）？不得使用流加密（RC4/AES-CFB），流加密的延展性允许审查系统实施选择密文攻击：发送 256 个变体，观察哪个不被拒绝 |
| **密文完整性验证** | 解密前是否验证 AEAD tag？跳过验证会接受篡改的密文 |
| **附加数据绑定** | AEAD 的 AD 是否绑定了足够的上下文（如地址信息、序列号）？ |
| **解密失败处理** | AEAD 解密失败时是否静默丢弃或安全回落？不得泄露解密失败的细节信息 |

### 2. 时间戳防重放

| 检查项 | 说明 |
|--------|------|
| **时间戳包含** | 协议载荷中是否包含客户端时间戳？ |
| **时间窗口校验** | 服务端是否校验时间戳与当前时间的偏差？推荐的窗口为 30 秒 |
| **时钟偏移容忍** | 时间窗口是否容忍合理的服务端-客户端时钟偏差？过小的窗口会误拒合法请求 |
| **延迟重放防御** | 即使时间戳在窗口内，是否仍检测同一 nonce 的重复使用？ |

### 3. Nonce 管理安全

| 检查项 | 说明 |
|--------|------|
| **Nonce 唯一性** | 同一密钥下每个 nonce 是否只使用一次？AEAD 的安全性依赖于 nonce 唯一性 |
| **Nonce 递增方向** | 两个方向的 nonce 是否独立计数？ |
| **Nonce 溢出处理** | 序列号接近上限时是否拒绝新连接而非回绕？ |
| **UDP nonce 处理** | UDP 场景是否使用显式 nonce？不得依赖隐式递增（UDP 无序）|
| **Nonce 记忆持久性** | Nonce 记忆池的生命周期是否足够长以覆盖延迟重放窗口（数天）？仅在单次会话内检测 nonce 重复无法防御延迟重放 |
| **Nonce 空间大小** | Nonce 空间是否足够大以避免碰撞？过小的 nonce 空间在高并发下可能产生碰撞 |

### 4. 选择密文攻击防御

| 检查项 | 说明 |
|--------|------|
| **拒绝篡改密文** | AEAD 解密失败时是否立即终止连接？不得尝试"部分接受" |
| **侧信道沉默** | 解密失败后的行为是否与正常处理一致（时间、资源使用）？ |
| **无错误细节** | 错误响应是否不包含解密失败的原因？ |
| **连接清理** | 认证失败后是否彻底清理连接状态？残留状态可能被后续探针利用 |
| **256 变体攻击** | 审查系统可能对捕获包逐字节变异后发送 256 个变体。AEAD 认证失败的一致性（所有变体都失败且响应相同）可以阻止此类攻击 |

### 5. 首包认证

| 检查项 | 说明 |
|--------|------|
| **密码哈希** | 首包的密码是否为哈希（如 SHA224）而非明文？明文密码可被直接重放 |
| **首包完整性** | 首包长度是否足够包含完整认证信息？过短的首包不应触发认证逻辑 |
| **地址验证** | 首包中的目标地址是否经过验证？格式错误的地址应静默拒绝 |
| **认证失败静默** | 密码不匹配时是否伪装为正常服务器？不得直接断开 |
| **首包与时戳绑定** | 首包认证数据是否包含时间戳或序列号？无时间维度的认证数据可被无限期重放 |

### 6. 密码认证安全

| 检查项 | 说明 |
|--------|------|
| **精确读取** | 是否分步精确读取用户名和密码？不得预读固定最大长度（阻塞/过度读取）|
| **凭据哈希** | 密码是否哈希后比较？ |
| **认证失败响应** | 认证失败是否返回标准协议拒绝响应？ |
| **缓冲区边界** | 认证缓冲区是否足够大？是否检查边界？ |

### 7. 重放响应行为分析防御

| 检查项 | 说明 |
|--------|------|
| **响应长度一致性** | 合法请求和重放请求的响应长度是否做了混淆？如果重放被拒绝时响应长度明显不同于正常代理响应，审查系统可通过长度差异推断代理存在 |
| **拒绝行为模拟** | 重放被检测到后，服务端是否模拟标准 Web 服务器的行为（如返回正常网页），而非返回空响应或断开？ |
| **时间侧信道** | 重放检测的时间开销是否与正常请求处理一致？显著的时间差异也可能被利用 |
| **响应内容模拟** | 重放被拒绝后的响应是否包含合理的 Web 内容（如 HTML 页面）？空响应或固定长度的短响应本身是可疑的 |

## 审计流程

1. **识别认证路径**: 确定协议的认证入口点和认证流程
2. **绘制攻击面**: 列出审查者可能发送的所有探针类型（即时重放、延迟重放、篡改重放、256 变体攻击、乱码）
3. **验证时间戳防御**: 确认时间戳校验窗口合理且覆盖所有认证路径
4. **验证 Nonce 防御**: 确认 nonce 唯一性保证、溢出处理、记忆持久性
5. **测试延迟重放**: 模拟数小时/数天后的重放攻击，验证服务器响应行为
6. **测试选择密文**: 确认 AEAD 对逐字节变异的密文有一致的拒绝行为
7. **测试认证失败路径**: 确认所有认证失败路径都不泄漏代理信息，响应行为与标准 Web 服务器一致

## 常见反模式（禁止）

```cpp
// ❌ 流加密无认证 — 选择密文攻击
// 审查者发送 256 个连接，每个改变一个字节
// 某个连接碰巧使得服务端不立即断开 → 确认是代理
aes_cfb_encrypt(key, nonce, plaintext);

// ✅ AEAD 认证加密 — 密文完整性保证
aead_context ctx(aead_cipher::aes_128_gcm, key);
ctx.seal(out, plaintext, ad);

// ❌ 固定读取最大长度 — 阻塞或过度读取
remaining = ulen + 1 + 255;  // 即使密码只有 8 字节
co_await recv_impl(buffer(remaining), ec);

// ✅ 分步精确读取
co_await recv_impl(buffer(ulen + 1), ec);  // UNAME + PLEN
auto plen = buf[2 + ulen];
co_await recv_impl(buffer(plen), ec);       // 仅读取实际密码长度

// ❌ 认证失败直接断开 — 暴露代理
if (!authenticated) {
    close();
}

// ✅ 认证失败返回标准错误或回落到真实 Web 服务
if (!authenticated) {
    co_await fallback_to_dest(raw_request);
}

// ❌ Nonce 记忆仅在单次会话内 — 无法防御延迟重放
// 会话结束即清空 nonce 历史

// ✅ Nonce 记忆跨会话持久化（如基于时间窗口的滑动缓存）
// 覆盖审查系统可能的延迟重放窗口

// ❌ 重放被拒绝后返回空响应 — 行为异常
if (is_replay(nonce)) {
    co_return;  // 空响应，与正常 Web 服务器行为不一致
}

// ✅ 重放被拒绝后模拟标准 Web 服务器行为
if (is_replay(nonce)) {
    co_await fallback_to_dest(raw_request);  // 转发到真实网站
}
```
