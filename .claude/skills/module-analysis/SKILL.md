# Skill: Prism 模块深度分析

## 触发条件

**以下情况触发：**

1. ✅ 写入知识库新模块文档时
2. ✅ 重构模块时需要更新文档
3. ✅ 用户要求"分析模块"、"写模块文档"、"详细分析"
4. ✅ 发现模块文档缺失或过时

**不触发的情况：**

- ❌ 仅修改 bug（使用 bug-to-wiki 技能）
- ❌ 仅修改配置（使用配置文档）
- ❌ 用户说"先不分析"

## 分析目标

对每个模块进行深度分析，输出文档必须包含：

1. **模块职责** — 核心功能、边界定义
2. **关键类和接口** — 类名、职责、关键方法、设计原理
3. **文件结构** — 头文件、源文件、每个文件的职责
4. **模块依赖** — 依赖的模块、被依赖的模块、依赖关系图
5. **文件依赖** — 哪个文件依赖哪个文件、头文件包含关系
6. **数据流** — 请求处理流程、关键调用链、数据转换
7. **设计原理** — 为什么这样设计、关键决策、性能优化
8. **关键代码片段** — 核心逻辑、重要配置项

## 分组策略

### 分组原则

1. **按调用链分组** — 相关模块放在一起分析
2. **每组 3-5 个模块** — 避免单组过大导致分析不深入
3. **核心模块单独分析** — agent、session、pipeline 等核心模块需要详细分析

### 默认分组方案

**组1：核心会话链路**
```
agent → session → worker → listener → balancer → dispatch → loader
```
分析重点：前端如何接收连接、如何分发到 worker、会话如何管理、配置如何加载

**组2：协议处理流水线**
```
protocol → recognition → pipeline → resolve
```
分析重点：协议如何识别、如何解析、如何路由、DNS 如何解析

**组3：传输与伪装层**
```
channel → multiplex → stealth → crypto → memory → fault → exception → trace → transformer → outbound
```
分析重点：连接池如何管理、多路复用如何工作、伪装方案如何实现、加密如何工作、内存如何分配

### 自定义分组

用户可以指定：
- 分析哪些模块
- 分组方式
- 分析深度

## 分析流程

### 步骤 1：确认分析范围

```markdown
## 分析确认

1. 分析哪些模块？ {模块列表}
2. 分组方式？ {默认分组/自定义分组}
3. 分析深度？ {概要/详细/深度}
4. 输出路径？ {H:/{module}/detail.md}
```

### 步骤 2：读取源码

对每个模块：
1. 读取头文件目录 `include/prism/{module}/`
2. 读取源文件目录 `src/prism/{module}/`
3. 读取相关测试文件 `tests/{module}*.cpp`
4. 读取相关配置文件

### 步骤 3：分析模块

对每个模块分析：

**1. 模块职责**
- 核心功能是什么
- 边界定义（做什么、不做什么）
- 在整体架构中的位置

**2. 关键类和接口**
- 类名、继承关系
- 关键方法签名和职责
- 接口设计原理

**3. 文件结构**
- 头文件列表和职责
- 源文件列表和职责
- 文件间依赖关系

**4. 模块依赖**
- 依赖的模块列表（输入）
- 被依赖的模块列表（输出）
- 依赖关系图（ASCII）

**5. 文件依赖**
- 头文件包含关系
- 源文件链接关系
- 编译依赖链

**6. 数据流**
- 请求处理流程
- 关键调用链
- 数据转换过程
- 状态变化

**7. 设计原理**
- 为什么这样设计
- 关键设计决策
- 性能优化点
- 权衡取舍

**8. 关键代码片段**
- 核心逻辑代码
- 重要配置项
- 关键算法

### 步骤 4：生成文档

按格式生成模块文档：

```yaml
---
title: {模块名} 模块详细设计
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
type: module

---
```

### 步骤 5：写入知识库

1. 创建目录 `H:/{module}/`（如果不存在）
2. 写入 `H:/{module}/detail.md`
3. 更新 `H:/wiki/index.md`
4. 更新 `H:/wiki/log.md`

## 质量控制

### 文档质量检查

**每个文档必须满足：**

1. **完整性** — 包含所有 8 个必要部分
2. **准确性** — 技术细节正确，代码片段可编译
3. **深度性** — 不仅描述"是什么"，还要解释"为什么"
4. **可追溯性** — 有文件路径、类名、方法名
5. **可链接性** — 至少 5 个 wikilinks
6. **可理解性** — AI 读完能理解模块设计

### 防止知识库污染

**以下情况拒绝写入：**

1. ❌ 分析不深入（仅描述表面功能）
2. ❌ 缺少设计原理（没有解释"为什么"）
3. ❌ 依赖关系不完整（缺少上下游模块）
4. ❌ 数据流不清晰（没有调用链）
5. ❌ 文件依赖缺失（没有头文件包含关系）

## 常见错误

### 错误 1：仅描述表面功能

**错误做法：**
```markdown
## 模块职责

multiplex 模块负责多路复用。
```

**正确做法：**
```markdown
## 模块职责

### 核心功能
- 实现 smux/yamux 多路复用协议
- 在单个 TCP 连接上复用多个逻辑流
- 支持流控制和流量控制
- 支持流优先级和权重

### 边界定义
- **做什么：** 多路复用协议编解码、流管理、流量控制
- **不做什么：** 不处理 TLS 加密（由 channel 处理）、不处理协议识别（由 recognition 处理）
```

### 错误 2：缺少设计原理

**错误做法：**
```markdown
## 关键类和接口

### smux::frame

**职责：** smux 帧格式编解码
```

**正确做法：**
```markdown
## 关键类和接口

### smux::frame

**职责：** smux 帧格式编解码

**设计原理：**
- 使用固定 12 字节头 + 变长 payload，便于零拷贝解析
- 支持 4 种帧类型：SYN（建立流）、FIN（关闭流）、PSH（数据推送）、NOP（心跳）
- 版本字段预留 1 字节，支持未来协议升级
```

### 错误 3：依赖关系不完整

**错误做法：**
```markdown
## 模块依赖

multiplex 依赖 channel。
```

**正确做法：**
```markdown
## 模块依赖

### 依赖的模块（输入）

| 模块 | 依赖原因 | 关键接口 |
|------|----------|----------|
| channel | 获取 TCP 连接 | `connection::dial()`, `connection::pool()` |
| memory | PMR 内存分配 | `memory::vector<T>`, `memory::string` |

### 被依赖的模块（输出）

| 模块 | 依赖原因 | 提供的接口 |
|------|----------|------------|
| pipeline | 协议流水线需要多路复用 | `multiplex::open_stream()`, `multiplex::accept_stream()` |
| protocol | 协议处理需要流接口 | `stream::read()`, `stream::write()` |

### 依赖关系图

```
pipeline ──→ [multiplex] ──→ channel
    ↑              ↑
    │              │
protocol        memory
```
```

### 错误 4：数据流不清晰

**错误做法：**
```markdown
## 数据流

数据从 pipeline 进入 multiplex，然后发送到 channel。
```

**正确做法：**
```markdown
## 数据流

### 请求处理流程

```
1. pipeline 调用 multiplex::open_stream()
2. multiplex 分配 stream_id
3. multiplex 构造 SYN 帧
4. multiplex 调用 channel::send() 发送 SYN 帧
5. 等待对端 ACK
6. 返回 stream 对象给 pipeline
7. pipeline 通过 stream::send() 发送数据
8. multiplex 构造 PSH 帧
9. multiplex 调用 channel::send() 发送 PSH 帧
```

### 关键调用链

```
pipeline::process()
    → multiplex::open_stream()
        → smux::craft::syn(stream_id)
        → channel::send(frame)
        → wait_for_ack()
    → stream::send(data)
        → smux::craft::psh(stream_id, data)
        → channel::send(frame)
```

### 数据转换

| 阶段 | 输入 | 输出 | 转换逻辑 |
|------|------|------|----------|
| 打开流 | stream 请求 | SYN 帧 | 添加 12 字节头 + stream_id |
| 发送数据 | 应用数据 | PSH 帧 | 添加 12 字节头 + stream_id + 数据 |
| 接收帧 | 字节流 | 帧对象 | 解析 12 字节头 + 提取 payload |
```
