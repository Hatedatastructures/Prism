# Skill: 密码学协议安全审计

在修改涉及加密、密钥交换、密钥派生、证书处理代码后，必须对变更部分执行以下审计清单。密码学实现的任何缺陷都可能导致整个防御体系崩溃。

## 审查系统的检测原理

审查系统在密码学层面有两个检测入口：

1. **证书链验证**：旁路设备实时检查服务端返回的证书 — 自签名证书直接阻断，CN/SAN 与 SNI 不匹配则封锁，非受信 CA 签发的证书标记为可疑。大量域名批量申请免费 CA 证书也会触发降级审查。
2. **密钥派生推断**：如果密钥派生存在缺陷（如 nonce 重复、密钥复用），审查系统可以通过选择密文攻击推断出加密密钥，进而解密整个会话。

此外，密码学实现的**时序侧信道**也需关注：如果密钥比较或签名验证不是常量时间的，审查系统可以通过精确测量响应时间来逐字节推断密钥。

## 触发条件

- 修改了 AEAD、HKDF、X25519、BLAKE3 等加密原语代码
- 修改了密钥交换或密钥派生逻辑
- 修改了 HMAC 认证逻辑
- 修改了帧加密/解密逻辑
- 修改了协议级加密（如 SS2022）
- 修改了配置中密钥相关字段（private_key、password、users）
- 修改了证书生成或签名逻辑

## 审计清单

### 1. AEAD 加密解密

| 检查项 | 说明 |
|--------|------|
| **Nonce 不重复** | 同一密钥下每次 seal/open 是否使用不同的 nonce？AEAD nonce 重复会导致密钥流复用，彻底破坏机密性 |
| **Nonce 递增安全** | nonce 递增溢出时是否正确拒绝？溢出后继续使用会导致 nonce 重复 |
| **Tag 长度验证** | 密文长度是否 >= tag_length()？open_size() 是否处理了密文短于 tag 的情况？ |
| **附加数据（AD）完整性** | AD 是否包含足够的上下文信息（记录头、序列号）？AD 缺失或不足会降低认证强度 |
| **密钥长度匹配** | 密钥长度是否与算法匹配？AES-128-GCM = 16 字节，AES-256-GCM = 32 字节，ChaCha20 = 32 字节 |
| **密钥零化** | 不再使用的密钥材料是否被安全擦除？密钥残留在内存中可能被内存取证获取 |

### 2. 密钥交换

| 检查项 | 说明 |
|--------|------|
| **低阶点检测** | 共享密钥计算后是否检查全零？（低阶点攻击）|
| **密钥对生成** | 是否使用密码学安全随机数生成器生成密钥对？|
| **静态密钥安全** | 静态 private_key 是否通过安全通道传递？不得在日志中输出 |
| **临时密钥使用** | 是否使用临时密钥对提供前向安全性？ |
| **密钥交换验证** | 共享密钥是否经过密钥确认步骤？确保双方派生出相同的密钥 |

### 3. HKDF 密钥派生

| 检查项 | 说明 |
|--------|------|
| **Extract 阶段** | HKDF-Extract 的 salt 和 IKM 是否正确？ |
| **Expand 阶段** | HKDF-Expand 的 info 字段是否包含足够的协议上下文？|
| **派生链完整性** | 从 shared_secret 到最终的应用密钥，派生链是否完整？是否遗漏了中间步骤？ |
| **输出长度** | 派生输出长度是否与下游算法匹配？32 字节密钥不应派生出 16 字节 |
| **派生上下文绑定** | 每一步派生的 info/label 是否绑定了足够的会话上下文（如握手摘要）？防止跨会话密钥碰撞 |

### 4. 常量时间操作

| 检查项 | 说明 |
|--------|------|
| **密码比较** | 密码/token 比较是否使用常量时间函数（如 `CRYPTO_memcmp`）？标准 `memcmp` 的时间差可被侧信道利用：审查系统通过精确测量响应时间来逐字节推断密钥 |
| **短标识比较** | 协议中的短标识符（如 short_id）匹配是否使用常量时间比较？ |
| **HMAC 验证** | HMAC 结果比较是否在常量时间内完成？ |
| **分支不可预测** | 密码学关键路径的条件分支是否不依赖于秘密数据？ |

### 5. 密钥生命周期

| 检查项 | 说明 |
|--------|------|
| **密钥隔离** | 不同用途的密钥（auth_key、hskey、appkey）是否从不同的派生路径生成？不得复用同一密钥。密钥复用会打破安全证明 — 不同上下文中的同一密钥可能被交叉攻击 |
| **密钥不跨会话** | 每个会话是否使用独立的密钥？密钥跨会话复用会破坏前向安全性 |
| **序列号空间** | 每个方向的序列号空间是否独立？读序列号和写序列号不得共享 |
| **密钥材料所有权** | 密钥数据是否由拥有所有权的容器持有？不得用 span 引用局部变量（悬垂引用 bug） |
| **密钥轮换** | 长期密钥是否有轮换机制？永久不变的密钥增加了被破解后的影响范围 |

### 6. 证书与签名

| 检查项 | 说明 |
|--------|------|
| **证书签发方式** | 审查系统直接阻断自签名证书。伪造证书必须能通过证书链验证（要么由受信 CA 签发，要么伪造得足够完整）。纯自签名在审查环境下不可用 |
| **伪造证书质量** | 伪造证书是否看起来足够真实？证书扩展字段是否完整？ |
| **签名算法** | 签名使用 HMAC 而非真正的数字签名时，客户端能否通过共享密钥自行验证？ |
| **证书链构建** | 伪造证书的链结构是否完整？不完整的证书链会被审查系统识别 |
| **CN/SAN 一致性** | 伪造证书的 CN 是否使用了目标域名？硬编码的代理软件名称是已知泄漏风险 |
| **序列号随机性** | 证书序列号是否随机生成？固定值是可识别特征 |
| **有效期合理性** | 证书有效期是否与真实 CA 证书一致？异常短或异常长的有效期是可疑的 |
| **批量申请降级** | 大量域名批量申请免费 CA 证书会触发审查系统的降级审查。使用固定数量的合法域名更安全 |
| **证书透明度** | 注册的证书会被记入 CT Log，可被事后关联分析。伪造证书的注册模式应避免批量特征 |

## 审计流程

1. **绘制密钥派生图**: 从初始密钥材料到最终使用的密钥，绘制完整的派生链
2. **验证每步参数**: 检查每一步 HKDF/AEAD 的 salt、IKM、info、nonce 是否正确
3. **检查常量时间**: 在所有涉及秘密数据比较的地方确认使用常量时间函数
4. **验证密钥隔离**: 确认不同用途的密钥不会碰撞或复用，派生路径的 info 互不相同
5. **验证证书链**: 确认证书能通过审查系统的链验证（非自签名、CN/SAN 与 SNI 一致、有效期合理）
6. **测试边缘场景**: 全零共享密钥、nonce 溢出、密钥长度不匹配等

## 常见反模式（禁止）

```cpp
// ❌ Nonce 重复 — 彻底破坏 AEAD 安全性
// 同一密钥 + 同一 nonce = 密钥流复用

// ❌ 非常量时间密码比较 — 侧信道泄露
// 审查系统通过精确测量响应时间逐字节推断密钥
if (std::string_view(received) == expected_password) { ... }

// ✅ 常量时间比较
if (CRYPTO_memcmp(received.data(), expected.data(), len) == 0) { ... }

// ❌ span 引用局部变量 — 密钥数据悬垂
std::string decoded = base64_decode(key);
result.decoded_privkey = std::span<const uint8_t>(
    reinterpret_cast<const uint8_t*>(decoded.data()), decoded.size());
// decoded 析构后 span 悬挂

// ✅ 拥有所有权的容器
result.decoded_privkey.assign(
    reinterpret_cast<const uint8_t*>(decoded.data()),
    reinterpret_cast<const uint8_t*>(decoded.data() + decoded.size()));

// ❌ 未检测低阶点
auto [ec, shared] = x25519(privkey, pubkey);
// shared 可能全零（低阶点攻击）

// ✅ 检测全零共享密钥
auto [ec, shared] = x25519(privkey, pubkey);
if (std::all_of(shared.begin(), shared.end(), [](auto b) { return b == 0; })) {
    return {fault::code::kexfail, {}};
}

// ❌ 密钥派生路径缺少上下文绑定
auto key = hkdf_expand(secret, "");  // 空 info，跨会话可碰撞

// ✅ 每步派生绑定会话上下文
auto key = hkdf_expand(secret, "proxy-session|" + session_id);
```
