# Skill: 流量统计分析防御审计

在修改涉及多路复用、填充、传输层、隧道转发代码后，必须对变更部分执行以下审计清单。审查系统通过宏观统计特征（熵值、包长、时间间隔、流量进出比、连接模式）训练机器学习模型来识别代理流量。

## 审查系统的检测原理

审查系统在宏观层面运行**多维度统计分类器**，核心逻辑不是"找代理特征"而是"判定流量是否像正常 Web 浏览"：

1. **熵值分类**：计算 TLS 记录的字节 popcount（每字节中 1 的位数）。若平均值落在 3.4-4.6 之间（近似纯随机），且不匹配已知 TLS/HTTP 握手特征、不以明文协议豁免，则判定为未知加密协议并阻断。关键洞察：**纯随机本身就是异常信号** — 正常 HTTPS 流量由于 TLS 记录头、HTTP 帧结构等明文字段的存在，熵值并不完美均匀。
2. **ML 包长/间隔分类**：提取连接前 N 个包的长度序列和间隔时间向量，用机器学习（随机森林、SVM）分类。关键特征包括：上行包均值、下行包方差、心跳包占比、IAT 平均值。
3. **首包指纹**：握手后第一个应用层包的长度和格式是高信息量特征。固定长度的认证字段 + 地址字段构成可训练的特征向量。
4. **宏观行为画像**：IP 级别的流量进出比、并发连接数、连接建立频率。代理节点与普通网站的流量模式截然不同。
5. **多会话关联分析**：审查系统不只看单个连接，而是关联同一客户端 IP 到同一服务器 IP 的所有连接模式。正常浏览行为是间歇性的（浏览 → 阅读 → 浏览），代理行为是持续性的（长时间稳定数据流）。多个短连接按时间排列构成的"连接节奏"也是分类特征。
6. **时间维度特征**：审查系统会提取流量在一天中的时间分布模式。正常用户有明显的昼夜规律，而代理流量通常全天均匀分布。
7. **DNS 关联**：审查系统不仅分析 TLS 流量，还会关联同一客户端 IP 的 DNS 查询模式。大量查询未知/新注册域名后立即连接特定 IP，这个 DNS→TLS 关联模式本身就是代理特征。
8. **历史行为基线**：审查系统为每个 IP 建立历史行为基线。如果某个 IP 的流量模式突然从"正常浏览"转变为"长连接代理模式"，基线偏移会触发告警。

## 触发条件

- 修改了多路复用协议代码
- 修改了填充机制代码
- 修改了传输层读写行为
- 修改了隧道双向转发逻辑
- 修改了加密封装逻辑
- 修改了帧大小、缓冲区大小相关配置
- 修改了连接管理、并发控制相关配置

## 审计清单

### 1. 熵值分布审计

| 检查项 | 说明 |
|--------|------|
| **防御目标明确** | 理解审查系统的熵值分类器：它检测的是"纯随机"模式（popcount 3.4-4.6），防御目标不是让熵更高，而是让流量**不呈现纯随机特征** |
| **填充数据随机性** | 填充帧的填充数据是否使用密码学安全随机数？全零填充使熵值骤降 |
| **明文特征保留** | 加密后的记录是否保留了足够的明文结构（如 TLS 记录头、HTTP 帧边界）？过多的明文结构暴露协议特征，过少则触发纯随机检测 |
| **帧头规律性** | 多路复用帧头是否暴露了固定字节模式？ |
| **首包 ASCII 豁免** | 审查系统对含有高比例可打印 ASCII 字符的流量会豁免放行（前 6 字节全为打印字符、前 50% 以上为 ASCII、或连续 20+ 可打印字符）。了解这个机制，但不要依赖它做防御（它也可能被反向利用） |

### 2. 包长分布（PLD）审计

| 检查项 | 说明 |
|--------|------|
| **首包大小** | 连接建立后的第一个数据包大小是否与正常 TLS 流量一致？过小的首包（如仅含 UUID）或固定长度的首包是已知代理特征 |
| **首包格式规律性** | 首包是否包含固定长度的认证字段 + 地址字段？这种固定格式可被 ML 训练提取。应通过随机填充破坏固定长度模式 |
| **请求/响应非对称性** | 正常 Web 流量呈强非对称性（小请求大响应），代理流量是否也保持这种分布？ |
| **TLS-in-TLS 消除** | 是否避免了 TLS 嵌套导致的叠加握手包特征？ |
| **帧分片边界** | 多路复用的帧边界是否导致密文在特定块对齐上呈现可预测的长度模式？ |
| **包长直方图** | 代理流量的包长直方图是否与正常 Web 浏览流量（集中在几个典型值如 MTU 附近）相似？高度碎片化的分布是异常特征 |
| **块对齐特征** | 加密后的数据是否呈现明显的块对齐模式（如总是 16 字节的倍数）？这种对齐模式在真实 HTTPS 流量中不存在 |

### 3. 包间隔时间（IAT）审计

| 检查项 | 说明 |
|--------|------|
| **keepalive 间隔规律性** | keepalive 是否以固定间隔发送？固定间隔是机器学习模型的强特征，应添加随机抖动 |
| **填充帧发送时机** | 填充帧是否与正常数据帧交替发送以混淆 IAT 分布？仅依赖周期性发送不够 |
| **响应延迟分布** | 代理请求的响应延迟是否与直连 Web 服务器有明显差异？ |
| **IAT 平均值** | ML 分类器会将 IAT 平均值作为特征。代理流量的 IAT 分布是否与正常浏览行为（长间隔的突发模式）一致？ |
| **心跳包占比** | 心跳包在总包数中的占比是否异常高？高心跳占比是隧道流量的典型特征 |
| **突发模式模拟** | 正常浏览是"读 → 点击 → 突发加载 → 读"的节奏。代理流量的 IAT 序列是否模拟了这种突发模式，而非均匀分布？ |

### 4. 首包特征审计

| 检查项 | 说明 |
|--------|------|
| **固定长度模式** | 首包是否为固定长度（如总是 56 字节）？固定长度本身就是高信息量特征，应通过随机填充使其长度在合理范围内波动 |
| **字段结构可识别性** | 首包内部是否有明显的字段分界（如密码 hash + 目标地址）？即使加密后，固定偏移处的字节分布也可能被统计提取 |
| **填充破坏模式** | 是否对首包进行随机长度的填充以破坏固定模式？填充长度应从足够大的范围内随机选择 |
| **多路复用掩护** | 是否可以通过多路复用将认证信息分散到多个帧中，而非集中在单个首包？ |

### 5. 多路复用流量整形

| 检查项 | 说明 |
|--------|------|
| **并发连接聚合** | 多路复用是否将多个并发请求聚合到单一底层连接中？"单 IP 爆发式多连接"行为本身就是翻墙特征 |
| **流 ID 分配规律** | stream_id 分配是否递增或有规律？可预测的 ID 分配可能被统计分析 |
| **帧交错模式** | 多个流的数据帧是否交错发送？顺序发送某一流的所有帧会暴露流边界 |
| **连接复用率** | 底层连接的生命周期和复用模式是否与正常浏览器行为一致？ |
| **流权重/优先级** | 如果使用 HTTP/2 多路复用，流的权重和依赖关系是否与正常浏览器的分配一致？所有流权重相同是简化实现的标志 |

### 6. 填充方案有效性

| 检查项 | 说明 |
|--------|------|
| **首包填充** | 填充方案是否对前 N 个包（高信息量握手包）进行强制填充？ |
| **填充长度随机性** | 填充长度是否从足够大的范围内随机选择？过小的范围不足以掩盖原始包长分布 |
| **停止条件** | 填充的停止阈值是否合理？过小的停止值意味着后续包无填充保护 |
| **零长度帧语义** | 当填充方案返回零长度时，是否跳过发送而非发送空帧？ |
| **填充与流量融合** | 填充帧是否在时序上与正常数据帧融合？如果填充帧总是在数据帧之后立即发送（固定模式），则填充本身就变成了新的特征 |
| **填充开销合理性** | 填充带来的额外流量开销是否合理？过高的填充比例（如流量翻倍）会在 IP 级别造成新的异常特征（流量进出比失衡） |

### 7. 宏观流量行为审计

| 检查项 | 说明 |
|--------|------|
| **流量进出比** | 代理节点的上下行流量比例是否与正常 Web 服务器一致？普通网站下行大上行少；代理节点可能呈现相反模式。这是 IP 级别的宏观统计，即使协议伪装完美也会暴露 |
| **连接频率** | 单个 IP 到目标 IP 的连接建立频率是否异常？每秒上百次新建连接远超正常浏览器行为 |
| **并发连接数** | 单个 IP 到同一目标的并发连接数是否过高？数百个长连接本身就是翻墙特征 |
| **连接持续时间** | 代理连接的生命周期分布是否与正常浏览行为一致？异常长的连接持续时间是可疑的 |
| **流量时间分布** | 服务器 IP 的全天流量是否有明显的昼夜差异？正常网站有高峰低谷，全天均匀分布的流量模式是代理特征 |
| **流量峰值模式** | 正常网站的流量峰值与热门事件/时段相关。代理流量的峰值是否也呈现类似的"事件驱动"模式？无外部事件驱动的流量峰值是可疑的 |

### 8. 多会话关联分析防御

| 检查项 | 说明 |
|--------|------|
| **连接节奏** | 同一客户端 IP 到服务器 IP 的多个连接在时间上是否构成间歇性的"浏览节奏"？连续不断的稳定连接流是代理特征 |
| **会话间间隔** | 正常用户浏览网页时，两次页面加载之间有阅读间隔。代理流量的会话间间隔是否模拟了这种行为？ |
| **连接数量模式** | 正常浏览器的连接模式是"短时间内爆发多个连接 → 长时间静默 → 再次爆发"。代理的"稳定流"模式与之不同 |
| **方向比一致性** | 多个连接的上行/下行比是否一致？如果每个连接都是 1:1 的对称流量，这本身就是异常特征（正常浏览是非对称的） |
| **DNS→TLS 关联** | 审查系统会关联同一客户端 IP 的 DNS 查询和后续 TLS 连接。大量查询未知域名后立即连接特定 IP，这个关联模式是代理特征。如果代理客户端直接使用 IP 连接而无对应 DNS 查询，"无 DNS 的 TLS 连接"也是异常 |
| **SNI 多样性** | 正常用户访问多种不同网站（SNI 多样性高）。如果代理的所有连接都使用同一个 SNI（或极少数 SNI），SNI 单一性是异常特征 |

### 9. 传输层行为合规性

| 检查项 | 说明 |
|--------|------|
| **禁用 Nagle** | 是否启用了 TCP_NODELAY？Nagle 算法的缓冲会导致包长分布异常 |
| **写入合并策略** | 多次小写入是否合并为单次大写入？多次小写入会产生大量短包 |
| **读取缓冲区大小** | 读取缓冲区是否与正常 TLS 实现一致？过小的缓冲区会产生过多的短读取 |
| **partial write 处理** | `async_write_some` 返回部分写入时，是否循环写入直到完成？静默丢弃数据会导致连接状态不一致 |

## 审计流程

1. **识别变更范围**: 确定本次修改影响哪个传输层/多路复用协议
2. **绘制数据流图**: 从应用层写入到网络层发送，追踪数据经过的每一层
3. **标注长度特征点**: 在每一层标记数据长度的变化（+header、+padding、+tag）
4. **计算熵值**: 对变更后的流量计算 popcount 分布，验证不落在纯随机检测区间
5. **分析包长直方图**: 评估最终网络包的长度分布是否与正常流量相似
6. **检查时间模式**: 评估 keepalive、padding 的发送间隔是否有规律性特征
7. **评估宏观行为**: 估算部署后的流量进出比、并发连接数、全天流量分布是否在合理范围内
8. **多会话关联模拟**: 模拟多个连续连接的时间模式，评估是否构成可疑的"连接节奏"
9. **DNS 关联评估**: 如果涉及客户端行为，评估 DNS→TLS 关联是否暴露代理模式

## 常见反模式（禁止）

```cpp
// ❌ 固定间隔 keepalive — 机器学习模型可提取周期性特征
while (is_active()) {
    timer.expires_after(std::chrono::milliseconds(fixed_interval));
    co_await timer.async_wait(...);
    co_await push_frame(command::nop, 0, {});
}

// ✅ 带随机抖动的 keepalive
while (is_active()) {
    auto jitter = random_in_range(-jitter_range, jitter_range);
    timer.expires_after(std::chrono::milliseconds(base_interval + jitter));
    co_await timer.async_wait(...);
    co_await push_frame(command::nop, 0, {});
}

// ❌ 零填充 — 全零字节使熵值骤降
memory::vector<std::uint8_t> waste_data(size, 0);

// ✅ 随机填充 — 熵值与正常加密流量一致
memory::vector<std::uint8_t> waste_data(size);
std::random_device rd;
std::generate(waste_data.begin(), waste_data.end(), [&]() { return rd(); });

// ❌ 固定长度首包 — ML 可提取的特征
co_await send(auth_hash + target_addr);

// ✅ 随机填充破坏固定模式
auto padding_len = random_in_range(0, max_padding);
co_await send(auth_hash + target_addr + random_bytes(padding_len));

// ❌ 填充帧总是在数据帧之后立即发送 — 填充本身变成了新特征
co_await send_data(payload);
co_await send_padding(random_len);  // 固定模式：数据→填充→数据→填充

// ✅ 填充帧的发送时机也需随机化
if (random_bool(padding_probability)) {
    co_await send_padding(random_len);
}
co_await send_data(payload);

// ❌ 客户端直接用 IP 连接 — 无 DNS 的 TLS 连接是异常
co_await connect(proxy_ip, 443);

// ✅ 通过域名解析获得 IP，审查系统看到的 DNS→TLS 关联更正常
auto ip = co_await resolve(cover_domain);
co_await connect(ip, 443);

// ❌ partial write 静默丢数据
written = co_await opts.to->async_write_some(data, ec);

// ✅ partial write 循环直到完成
auto remaining = data;
while (!remaining.empty()) {
    written = co_await opts.to->async_write_some(remaining, ec);
    if (ec) { co_return; }
    remaining = remaining.subspan(written);
}
```
